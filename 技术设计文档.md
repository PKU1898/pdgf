# 项目信息
项目名称：拼豆工坊
技术栈：
- 前端：Uni-app (Vue3 + Vite) + TypeScript + Tailwind CSS + Pinia
- 后端：Node.js (Express) + TypeScript + Prisma
- 数据库：PostgreSQL (通过 Prisma ORM 操作)
- 存储：腾讯云 COS (图片存储)
- 包管理：pnpm (前后端均使用)

# 常用命令
## 前端 (在 /miniapp 目录下)
- pnpm dev:mp-weixin    # 启动微信小程序开发模式
- pnpm build:mp-weixin  # 构建微信小程序生产版本

## 后端 (在 /server 目录下)
- pnpm dev               # 启动开发服务器 (热重载)
- pnpm build             # 编译 TypeScript 到 dist
- pnpm start             # 运行生产版本
- npx prisma studio      # 打开数据库可视化管理界面
- npx prisma migrate dev # 运行数据库迁移
- npx prisma generate    # 生成 Prisma Client

# 编码规范与红线（绝对不能违反）
- 前端组件必须使用 Vue3 `<script setup lang="ts">` 语法。
- 后端接口统一返回格式：`{ code: number, message: string, data: any }`。
- 数据库操作一律通过 `prisma client`，禁止手写 SQL 查询。
- 前端状态管理使用 Pinia，按模块划分 store。
- 所有异步操作必须使用 try-catch 捕获异常，禁止空 catch 块（至少 console.error）。
- 样式优先使用 Tailwind CSS 类名，严格使用设计规范中定义的 token (如 `bg-bg`, `text-primary`)。
- **红线1：禁止使用小程序 Canvas API 进行图纸画板的编辑渲染！** 必须使用 CSS Grid + Div 实现，只在最终“导出图片”时使用离屏 Canvas。
- **红线2：禁止在前端代码中暴露任何后端密钥**（COS SecretKey、JWT Secret等），必须走后端 API 代理。
- **红线3：禁止将大图直接读入 Node.js 内存进行像素遍历**，必须使用 `sharp` 库的流式处理和区域提取。
- **红线4：禁止提交 `.env` 文件到 Git**，只提供 `.env.example`。
- **红线5：禁止使用 `any` 类型**，如类型未知必须使用 `unknown` 并进行类型守卫。

## 用户与鉴权模块 (微信一键登录闭环)
├── task-006: 实现微信静默登录与手机号快捷登录接口 (支持一键登录)
│   ├── **目标**：实现微信用户无感静默登录及手机号一键绑定，生成 JWT Token。
│   ├── **交互流程**：前端调用 `wx.login` 获取 code -> 发送至后端静默登录接口 -> 若未绑定手机号，前端调用 `getPhoneNumber` 获取 code -> 发送至后端绑定接口 -> 返回最终 token。
│   ├── **技术实现**：
    │   1. API 1: `POST /api/auth/silent-login`。接收 `{ code: string }`。
    │      - 请求微信 `jscode2session` 获取 openid 和 session_key。
    │      - 在 User 表查找 openid，若不存在则创建（此时无手机号，vipStatus=0）。
    │      - 生成包含 userId 和 isPhoneVerified 状态的 token 返回。
    │   2. API 2: `POST /api/auth/bind-phone`。接收 `{ phoneCode: string }` (需鉴权)。
    │      - 调用微信 `getPhoneNumber` 接口，使用 session_key 解密获取真实手机号。
    │      - 更新 User 表的 phone 字段，修改 token 中 isPhoneVerified 为 true。
    │      - 返回新的 token。
    │   3. 使用 `jsonwebtoken` 生成 token，有效期 7 天。
│   ├── **安全要求**：JWT Secret 必须高强度。session_key 绝不能返回给前端。解密手机号必须在后端完成。校验微信接口的签名防止伪造。
│   ├── **性能要求**：接口响应时间 < 500ms。
│   ├── **正常场景**：
    │   - 老用户：静默登录直接返回完整 token，前端无弹窗。
    │   - 新用户：静默登录返回未绑定 token -> 弹出微信一键登录弹窗 -> 获取手机号 -> 绑定成功返回完整 token。
│   ├── **异常场景**：
    │   - 微信 code 无效/过期返回 `{ code: 1001, message: '微信登录失败' }`。
    │   - session_key 过期导致手机号解密失败，返回 `{ code: 1002, message: 'Session过期，请重新登录' }`。
    │   - 微信接口网络错误返回 500。
│   └── **测试目标**：
        - Given 有效的微信 code When 调用 silent-login Then 返回 200 及 token。
        - Given 未绑定手机号的 token 和有效 phoneCode When 调用 bind-phone Then 返回新 token 且数据库更新手机号。
        - Given 无效 phoneCode When 调用 bind-phone Then 返回 1002。
│
├── task-007: 实现后端 JWT 鉴权中间件，保护需登录的后端接口
│   ├── **目标**：保护后端 API 安全，拦截未授权或未绑定手机号的请求。
│   ├── **交互流程**：前端请求携带 token -> 中间件验证 -> 放行或拒绝。
│   ├── **技术实现**：
    │   1. 在 `/server/src/middleware/auth.ts`，从 `req.headers.authorization` 提取 Bearer token。
    │   2. 使用 `jwt.verify` 验证签名和有效期。成功则将 `userId` 和 `isPhoneVerified` 挂载到 `req.body`。
    │   3. 验证失败返回 401。
    │   4. 针对需要强制绑定手机号的接口（如导出、保存），若 `isPhoneVerified` 为 false，返回 403 `{ code: 4031, message: '请先绑定手机号' }`。
│   ├── **安全要求**：验证 Token 完整性，处理 Token 过期异常。区分未登录和未绑定手机号的状态码。
│   ├── **性能要求**：JWT 校验耗时 < 10ms。
│   ├── **正常场景**：有效 Token 请求顺利到达后续路由。
│   ├── **异常场景**：Token 缺失、格式错误、签名不匹配返回 401；未绑定手机号访问核心接口返回 4031。
│   └── **测试目标**：
        - Given 请求携带有效 Token When 经过中间件 Then req.body.userId 存在。
        - Given 无 Token When 请求 Then 返回 401。
        - Given 未绑定手机的 Token When 请求保存接口 Then 返回 4031。
│
├── task-008: 实现前端全局请求拦截器与微信一键登录 UI 闭环
│   ├── **目标**：前端请求自动化鉴权，实现微信原生一键登录体验。
│   ├── **交互流程**：
    │   1. App启动时调用 `wx.login` 获取 code，请求后端静默登录。
    │   2. 若接口返回需要绑定手机号，弹出底部半屏弹窗，包含微信原生 `<button open-type="getPhoneNumber">`。
    │   3. 用户点击一键登录，获取 phoneCode 发送至后端绑定。
    │   4. 绑定成功后更新本地 token，关闭弹窗。
    │   5. 请求拦截器自动注入 token，响应拦截器处理 401/4031。
│   ├── **技术实现**：
    │   1. 在 `/miniapp/src/api/request.ts` 封装 `uni.request`。
    │   2. 请求拦截：从 `uni.getStorageSync('token')` 读取并添加到 header。
    │   3. 响应拦截：若为 401 清理 token 并重启登录流程；若为 4031 触发弹出登录弹窗。
    │   4. 登录弹窗组件：使用微信原生按钮获取手机号，无感绑定。
│   ├── **安全要求**：Token 失效应立即清除本地存储。获取手机号必须由用户主动触发点击。
│   ├── **性能要求**：拦截器逻辑轻量，不增加明显请求延迟。登录弹窗动画 300ms。
│   ├── **正常场景**：老用户无感登录；新用户弹出一键登录弹窗，点击后秒级完成注册/登录。
│   ├── **异常场景**：用户拒绝授权手机号，停留在当前页面，提示"需绑定手机号以保存数据"。网络中断提示重试。
│   └── **测试目标**：
        - Given 本地存在 Token When 发起请求 Then Header 包含 Authorization。
        - Given 后端返回 4031 When 拦截器捕获 Then 弹出带微信一键登录按钮的弹窗。
        - Given 用户点击一键登录并授权 When 绑定成功 Then 弹窗关闭，原请求自动重试。
│
└── task-009: 搭建首页布局 (底部三个 TabBar：工作台、豆库、我的)
    ├── **目标**：搭建应用主框架，实现极简微信风首页 UI。
    ├── **交互流程**：底部切换 Tab -> 首页展示创建入口和最近列表 -> 点击创建拉起相册。

├── task-015: 实现后端图片像素化核心算法 (基于 sharp 库缩放图片至指定宽高提取像素)
│   ├── **目标**：将大图缩小并提取像素 RGB。
│   ├── **交互流程**：无。
│   ├── **技术实现**：

    │   2. 逻辑：使用 `sharp(buffer).resize(width, height, { fit: 'fill' }).raw().toBuffer()`。遍历返回的 Buffer，每 3 位 (RGB) 提取为一个像素对象，生成 `number[][][]` (高x宽x3) 二维数组返回。

│   ├── **正常场景**：返回正确的像素二维数组。
│   ├── **异常场景**：图片格式不支持、图片损坏抛出异常，由上层捕获。
│   └── **测试目标**：Given 5MB JPG 图片 When 调用 pixelateImage(100, 100) Then 返回 100x100x3 的数组。

├── task-016: 实现后端颜色映射算法 (计算欧氏距离，将任意 RGB 映射到最接近的品牌色号)
│   ├── **目标**：将任意 RGB 转换为最接近的拼豆色号 ID。
│   ├── **交互流程**：无。
│   ├── **技术实现**：
    │   1. 封装：在 `/server/src/utils/colorMapper.ts` 实现函数 `mapColors(pixelArray: number[][][], brandColors: BeadColor[])`。
    │   2. 算法：遍历像素，计算与目标品牌色卡 RGB 的欧氏距离: `Math.sqrt((r1-r2)**2 + (g1-g2)**2 + (b1-b2)**2)`。
    │   3. 结果：找出距离最小的色号，替换原像素，返回色号 ID 的二维数组 `string[][]` (如 `[["mard_001", "mard_002"], ...]`)。
│   ├── **安全要求**：无。
│   ├── **性能要求**：100x100 图纸映射耗时 < 500ms。超大规模 (>2000颗豆) 考虑异步任务。
│   ├── **正常场景**：返回映射后的色号数组。
│   ├── **异常场景**：色卡数据为空时抛出业务异常。
│   └── **测试目标**：Given 像素 RGB(255,0,0) 和 MARD 色卡 When 映射 Then 返回最接近红色的色号 ID。

# 编码规范与红线（绝对不能违反）
- 前端组件必须使用 Vue3 `<script setup lang="ts">` 语法。
- 后端接口统一返回格式：`{ code: number, message: string, data: any }`。
- 数据库操作一律通过 `prisma client`，禁止手写 SQL 查询。
- 前端状态管理使用 Pinia，按模块划分 store。
- 所有异步操作必须使用 try-catch 捕获异常，禁止空 catch 块（至少 console.error）。
- 样式优先使用 Tailwind CSS 类名，严格使用设计规范中定义的 token (如 `bg-bg`, `text-primary`)。
- **红线1：禁止使用小程序 Canvas API 进行图纸画板的编辑渲染！** 必须使用 CSS Grid + Div 实现，只在最终“导出图片”时使用离屏 Canvas。
- **红线2：禁止在前端代码中暴露任何后端密钥**（COS SecretKey、JWT Secret等），必须走后端 API 代理。
- **红线3：禁止将大图直接读入 Node.js 内存进行像素遍历**，必须使用 `sharp` 库的流式处理和区域提取。
- **红线4：禁止提交 `.env` 文件到 Git**，只提供 `.env.example`。
- **红线5：禁止使用 `any` 类型**，如类型未知必须使用 `unknown` 并进行类型守卫。

## 用户与鉴权模块 (微信一键登录闭环)
├── task-006: 实现微信静默登录与手机号快捷登录接口 (支持一键登录)
│   ├── **目标**：实现微信用户无感静默登录及手机号一键绑定，生成 JWT Token。
│   ├── **交互流程**：前端调用 `wx.login` 获取 code -> 发送至后端静默登录接口 -> 若未绑定手机号，前端调用 `getPhoneNumber` 获取 code -> 发送至后端绑定接口 -> 返回最终 token。
│   ├── **技术实现**：
    │   1. API 1: `POST /api/auth/silent-login`。接收 `{ code: string }`。
    │      - 请求微信 `jscode2session` 获取 openid 和 session_key。
    │      - 在 User 表查找 openid，若不存在则创建（此时无手机号，vipStatus=0）。
    │      - 生成包含 userId 和 isPhoneVerified 状态的 token 返回。
    │   2. API 2: `POST /api/auth/bind-phone`。接收 `{ phoneCode: string }` (需鉴权)。
    │      - 调用微信 `getPhoneNumber` 接口，使用 session_key 解密获取真实手机号。
    │      - 更新 User 表的 phone 字段，修改 token 中 isPhoneVerified 为 true。
    │      - 返回新的 token。
    │   3. 使用 `jsonwebtoken` 生成 token，有效期 7 天。
│   ├── **安全要求**：JWT Secret 必须高强度。session_key 绝不能返回给前端。解密手机号必须在后端完成。校验微信接口的签名防止伪造。
│   ├── **性能要求**：接口响应时间 < 500ms。
│   ├── **正常场景**：
    │   - 老用户：静默登录直接返回完整 token，前端无弹窗。
    │   - 新用户：静默登录返回未绑定 token -> 弹出微信一键登录弹窗 -> 获取手机号 -> 绑定成功返回完整 token。
│   ├── **异常场景**：
    │   - 微信 code 无效/过期返回 `{ code: 1001, message: '微信登录失败' }`。
    │   - session_key 过期导致手机号解密失败，返回 `{ code: 1002, message: 'Session过期，请重新登录' }`。
    │   - 微信接口网络错误返回 500。
│   └── **测试目标**：
        - Given 有效的微信 code When 调用 silent-login Then 返回 200 及 token。
        - Given 未绑定手机号的 token 和有效 phoneCode When 调用 bind-phone Then 返回新 token 且数据库更新手机号。
        - Given 无效 phoneCode When 调用 bind-phone Then 返回 1002。
│
├── task-007: 实现后端 JWT 鉴权中间件，保护需登录的后端接口
│   ├── **目标**：保护后端 API 安全，拦截未授权或未绑定手机号的请求。
│   ├── **交互流程**：前端请求携带 token -> 中间件验证 -> 放行或拒绝。
│   ├── **技术实现**：
    │   1. 在 `/server/src/middleware/auth.ts`，从 `req.headers.authorization` 提取 Bearer token。
    │   2. 使用 `jwt.verify` 验证签名和有效期。成功则将 `userId` 和 `isPhoneVerified` 挂载到 `req.body`。
    │   3. 验证失败返回 401。
    │   4. 针对需要强制绑定手机号的接口（如导出、保存），若 `isPhoneVerified` 为 false，返回 403 `{ code: 4031, message: '请先绑定手机号' }`。
│   ├── **安全要求**：验证 Token 完整性，处理 Token 过期异常。区分未登录和未绑定手机号的状态码。
│   ├── **性能要求**：JWT 校验耗时 < 10ms。
│   ├── **正常场景**：有效 Token 请求顺利到达后续路由。
│   ├── **异常场景**：Token 缺失、格式错误、签名不匹配返回 401；未绑定手机号访问核心接口返回 4031。
│   └── **测试目标**：
        - Given 请求携带有效 Token When 经过中间件 Then req.body.userId 存在。
        - Given 无 Token When 请求 Then 返回 401。
        - Given 未绑定手机的 Token When 请求保存接口 Then 返回 4031。
│
├── task-008: 实现前端全局请求拦截器与微信一键登录 UI 闭环
│   ├── **目标**：前端请求自动化鉴权，实现微信原生一键登录体验。
│   ├── **交互流程**：
    │   1. App启动时调用 `wx.login` 获取 code，请求后端静默登录。
    │   2. 若接口返回需要绑定手机号，弹出底部半屏弹窗，包含微信原生 `<button open-type="getPhoneNumber">`。
    │   3. 用户点击一键登录，获取 phoneCode 发送至后端绑定。
    │   4. 绑定成功后更新本地 token，关闭弹窗。
    │   5. 请求拦截器自动注入 token，响应拦截器处理 401/4031。
│   ├── **技术实现**：
    │   1. 在 `/miniapp/src/api/request.ts` 封装 `uni.request`。
    │   2. 请求拦截：从 `uni.getStorageSync('token')` 读取并添加到 header。

│   ├── **安全要求**：Token 失效应立即清除本地存储。获取手机号必须由用户主动触发点击。
│   ├── **性能要求**：拦截器逻辑轻量，不增加明显请求延迟。登录弹窗动画 300ms。
│   ├── **正常场景**：老用户无感登录；新用户弹出一键登录弹窗，点击后秒级完成注册/登录。
│   ├── **异常场景**：用户拒绝授权手机号，停留在当前页面，提示"需绑定手机号以保存数据"。网络中断提示重试。
│   └── **测试目标**：
        - Given 本地存在 Token When 发起请求 Then Header 包含 Authorization。
        - Given 后端返回 4031 When 拦截器捕获 Then 弹出带微信一键登录按钮的弹窗。
        - Given 用户点击一键登录并授权 When 绑定成功 Then 弹窗关闭，原请求自动重试。
│

│   ├── **目标**：将大图缩小并提取像素 RGB。
│   ├── **交互流程**：无。

│   ├── **异常场景**：图片格式不支持、图片损坏抛出异常，由上层捕获。
│   └── **测试目标**：Given 5MB JPG 图片 When 调用 pixelateImage(100, 100) Then 返回 100x100x3 的数组。

├── task-016: 实现后端颜色映射算法 (计算欧氏距离，将任意 RGB 映射到最接近的品牌色号)
│   ├── **目标**：将任意 RGB 转换为最接近的拼豆色号 ID。
│   ├── **交互流程**：无。
│   ├── **技术实现**：
    │   1. 封装：在 `/server/src/utils/colorMapper.ts` 实现函数 `mapColors(pixelArray: number[][][], brandColors: BeadColor[])`。
    │   2. 算法：遍历像素，计算与目标品牌色卡 RGB 的欧氏距离: `Math.sqrt((r1-r2)**2 + (g1-g2)**2 + (b1-b2)**2)`。
    │   3. 结果：找出距离最小的色号，替换原像素，返回色号 ID 的二维数组 `string[][]` (如 `[["mard_001", "mard_002"], ...]`)。
│   ├── **安全要求**：无。
│   ├── **性能要求**：100x100 图纸映射耗时 < 500ms。超大规模 (>2000颗豆) 考虑异步任务。
│   ├── **正常场景**：返回映射后的色号数组。
│   ├── **异常场景**：色卡数据为空时抛出业务异常。
│   └── **测试目标**：Given 像素 RGB(255,0,0) 和 MARD 色卡 When 映射 Then 返回最接近红色的色号 ID。
